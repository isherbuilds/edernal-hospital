# Arogya OS Drizzle Schema Pack

Production-oriented Drizzle/Postgres schema baseline for the Arogya OS HMS MVP.

This pack is intentionally broader than a first sprint implementation. It gives engineering a single typed database contract for Phase 0 and early private beta while preserving the system-design decisions: FHIR-native canonical store, hot-path projections, tenant isolation, audit/provenance, human AI sign-off, event outbox, integrations, offline edge sync, onboarding/migration and Better Auth-managed identity.

## Install

```bash
pnpm install
cp .env.example .env # create DATABASE_URL yourself
pnpm db:generate
pnpm db:migrate
```

## Better Auth workflow

`src/db/schema/auth.generated.ts` is a committed schema snapshot for the plugin posture in `src/lib/auth/auth.example.ts`:

- core Better Auth tables: user, session, account, verification
- admin fields
- organization + member + invitation + team/teamMember
- two-factor auth
- passkeys
- phone number OTP
- API keys
- JWT/JWKS
- OAuth provider tables

When the Better Auth config changes, regenerate the auth schema and compare the diff:

```bash
pnpm auth:generate
pnpm db:generate
```

Keep Arogya-specific identity/RBAC tables in `iam.ts`; do not hand-edit Better Auth-owned columns unless you intentionally diverge from the generated schema.

## RLS model

The schema is RLS-ready. Run SQL files in this order before/with initial migrations:

```bash
psql "$DATABASE_URL" -f drizzle/sql/001_extensions.sql
psql "$DATABASE_URL" -f drizzle/sql/002_rls_context.sql
pnpm db:migrate
psql "$DATABASE_URL" -f drizzle/sql/003_rls_policies.sql
```

Every request transaction should set context:

```sql
select app.set_request_context(
  p_tenant_id := '00000000-0000-0000-0000-000000000000'::uuid,
  p_facility_id := null,
  p_user_id := 'better-auth-user-id',
  p_purpose_of_use := 'treatment'
);
```

Use `p_bypass_rls := true` only from locked-down maintenance roles and only for migration/offboarding/support jobs with separate audit events.

## Implementation notes

1. Treat `fhir_resource_versions` as canonical clinical truth. Projection tables exist for fast UX and reporting.
2. Never publish raw PHI in outbox payloads unless the consumer explicitly requires it and the topic is PHI-classified.
3. Real PHI should not enter dev. Use synthetic seed data and tenant-scoped staging only.
4. Keep clinical write finalization behind state machines and `ai_signoff_guards`.
5. Use `audit_events` for security/compliance audit. Do not confuse it with operational logs or analytics.
6. Use `fhir_provenance_records` for AI and clinician authorship/sign-off traceability.
7. Use `idempotency_keys`, `outbox_events`, `inbox_processed_messages`, and integration message tables consistently for retries and re-drive.
8. Keep config-pack tables generic; India-specific GST, ABHA, ABDM, payer and terminology behavior should remain data/config, not hardcoded branches.

## What this pack does not do

- It does not replace formal FHIR profile validation.
- It does not produce a legal compliance certification by itself.
- It does not include seed data, migrations generated by Drizzle Kit, or cloud/IaC resources.
- It does not assert that every Better Auth plugin import path will remain stable forever; regenerate against your pinned Better Auth version.
